6. Team Topologies Framework (Applicable to Case)

Core Concepts for Capability Collision Problem

Team Type Role HD Application
Stream-aligned End-to-end product ownership Magic Apron, Recommendations, Search, etc.
Platform Shared capabilities, reduces cognitive load Proposed "Event Streaming Layer"
Enabling Helps teams adopt new practices Governance/architecture teams
Complicated-subsystem Deep specialization Inventory system, Pricing engine

Interaction Modes

Mode Description When to Use
Collaboration High-bandwidth, high-cost joint work Building new shared capabilities
X-as-a-Service Clear boundaries, low coordination Steady-state platform consumption
Facilitating Temporary help removing obstacles Onboarding teams to new patterns

Key Insight: Platform teams should aim for X-as-a-Service as default mode (low coordination overhead) while using Collaboration mode only when developing new capabilities.

Case Application: HD's collision problem = too much implicit collaboration without explicit interfaces. Solution = create explicit platform (event stream) with X-as-a-Service consumption model.

Source: Team Topologies, Martin Fowler, IT Revolution

8. Amazon Two-Pizza Teams Model

Organizational Philosophy

Amazon's "two-pizza teams" (2PTs) represent an extreme divisional organization: - Size: 6-10 people (small enough to feed with two pizzas) - Autonomy: Teams are free to execute relatively autonomously to maximize their "fitness function" - Structure: Engineers, technical PMs, designers report directly to 2PT lead - Coordination: "No need to coordinate even across teams, let alone across divisions, to get something done"

Source: Jason Crawford, AWS Executive Insights

Governance Model

Amazon uses "guardrails" rather than "tollgates": - Guardrails = mechanisms that allow teams to run fast while making high-quality decisions - Tollgates = approval gates that impede speed

Case Relevance: HD's challenge is similar — how to provide guardrails (visibility, shared data) without tollgates (centralized approval).
